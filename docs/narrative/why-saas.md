# Why SaaS

Because it's the business model that matches how the underlying identity actually works: build a system once, let it serve many, keep improving it instead of rebuilding it per client. That's the opposite of the project-based consulting pattern that filled a lot of the 2008–2019 years — real systems thinking, but rebuilt from scratch for every engagement instead of compounding.

Multi-tenant SaaS architecture shows up as the throughline across the most recent roles for exactly this reason: Leasi's platform (inventory, equipment rental, maintenance, project/resource management, one codebase serving many tenants), Cognosic's curriculum platform for universities, ServiceBuddy's services marketplace. Different domains, same architectural shape — and each one an opportunity to get better at the shape itself, not just the domain.

SaaS also forces the multi-tenant, API-first, cloud-native discipline that happens to be exactly the skill set built over the Azure-and-integration years (2019–2025) — Azure Functions, Event Grid, Service Bus, API Management, the whole Azure iPaaS toolkit from the Cegal and Leasi work. SaaS isn't a new direction; it's what that decade of integration and platform work was always pointing toward.
